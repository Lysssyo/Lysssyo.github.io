# xv6 用户页表

## 一、用户页表概述

xv6 为每个进程维护一个独立的用户页表，只包含该进程用户内存的映射。用户页表**不是恒等映射**,每个进程以为自己的地址空间从 0 开始连续排列，但实际的物理页可能散布在内存各处。

页表实现了三个重要功能：

1. **私有内存**：不同进程的相同虚拟地址映射到不同物理页，互相看不到对方的数据
2. **连续的假象**：用户看到连续的虚拟地址，实际物理页可以不连续
3. **共享 trampoline**：所有进程的用户页表都把 TRAMPOLINE 映射到同一个物理页

## 二、用户虚拟地址空间布局

```
MAXVA →        ┌──────────────┐
               │  trampoline  │  代码，所有进程共享，无 PTE_U
               ├──────────────┤
               │  trapframe   │  数据，每进程独立，无 PTE_U
               ├──────────────┤
               │              │
               │  （大片未映射）│
               │              │
p->sz          ├──────────────┤
               │    stack     │  用户栈，一页
               ├──────────────┤
               │  guard page  │  无映射，栈溢出时触发 page fault
               ├──────────────┤
               │    heap      │  向上增长（sbrk 扩展）
               ├──────────────┤
               │    data      │  全局变量等
               ├──────────────┤
               │    text      │  用户代码
 0 →           └──────────────┘
```

`p->sz` 标记已分配区域的顶部（text + data + heap + guard page + stack）。

用户栈的初始内容（由 `exec` 设置，从顶部往下）：

- 命令行参数的字符串（如 "ls"、"-l"）
- argv[] 数组（指向上面字符串的指针）
- argv 指针、argc、main 的返回地址

效果是程序启动时就像刚调用了 `main(argc, argv)` 一样。

## 三、用户页表的创建：proc_pagetable

```c
pagetable_t proc_pagetable(struct proc *p){
  pagetable_t pagetable;

  pagetable = uvmcreate();        // 分配空的根页表页
  if(pagetable == 0)
    return 0;

  // 映射 trampoline（无 PTE_U，用户态不能访问）
  if(mappages(pagetable, TRAMPOLINE, PGSIZE,
              (uint64)trampoline, PTE_R | PTE_X) < 0){
    uvmfree(pagetable, 0);
    return 0;
  }

  // 映射 trapframe（无 PTE_U，用户态不能访问）
  if(mappages(pagetable, TRAPFRAME, PGSIZE,
              (uint64)(p->trapframe), PTE_R | PTE_W) < 0){
    uvmunmap(pagetable, TRAMPOLINE, 1, 0);
    uvmfree(pagetable, 0);
    return 0;
  }

  return pagetable;
}

// create an empty user page table.
// returns 0 if out of memory.
pagetable_t uvmcreate(){
  pagetable_t pagetable;
  pagetable = (pagetable_t) kalloc();
  if(pagetable == 0)
    return 0;
  memset(pagetable, 0, PGSIZE);
  return pagetable;
}
```

`proc_pagetable` 只建立 trampoline 和 trapframe 两个特殊映射。用户程序的 text、data、stack 等映射在后续步骤中建立。

`proc_pagetable`在 `allocproc` 中被调用：`p->pagetable = proc_pagetable(p)`。

`allocproc`完整代码见：[`allocproc` 完整代码](030-%20xv6%20trampoline%20与%20trapframe.md#`allocproc`%20完整代码)

### 权限说明

- trampoline：`PTE_R | PTE_X`，可读可执行，**无 PTE_U**
- trapframe：`PTE_R | PTE_W`，可读可写，**无 PTE_U**

没有 `PTE_U` 意味着用户态代码访问会触发 page fault。只有 supervisor mode 可以访问（trampoline 代码在 trap 触发后以 supervisor 身份运行，虽然此时还在用户页表下）。

### 错误处理

按顺序分配资源，失败时回滚已分配的：

- 第一个映射（trampoline）失败：直接 `uvmfree` 释放页表
- 第二个映射（trapframe）失败：先 `uvmunmap` 清掉 trampoline 的 PTE，再 `uvmfree`

必须先清叶子 PTE 再释放页表，否则 `freewalk` 遇到有效叶子 PTE 会 panic。

## 四、用户内存的映射：分步建立

`proc_pagetable` 只是第一步，用户内存的映射取决于进程如何创建：

**通过 fork：**

```
allocproc()     →  空页表 + trampoline + trapframe
uvmcopy()       →  复制父进程的 text、data、stack 到子进程页表
```

**通过 fork + exec（最常见）：**

```
fork:
  allocproc()   →  空页表 + trampoline + trapframe
  uvmcopy()     →  复制父进程的用户内存

exec:
  创建新页表    →  trampoline + trapframe
  加载新程序    →  映射 text、data、guard page、stack
  丢弃旧页表
```

> [!TIP]
> exec 为什么创建新页表
> 
> `exec` 用新程序替换当前进程的内存。它不在旧页表上修改，而是创建新页表，加载成功后才丢弃旧页表。这是**原子替换**策略：如果加载到一半失败，旧页表不受影响，进程可以继续用旧程序运行。
> 

**第一个进程（init）：**

```
allocproc()     →  空页表 + trampoline + trapframe
userinit()      →  手动映射一小段初始代码
exec("init")    →  丢弃旧页表，创建新页表，加载 init 程序
```

## 五、用户内存的扩展：uvmalloc

```c
uint64 uvmalloc(pagetable_t pagetable, uint64 oldsz, uint64 newsz){
  char *mem;
  uint64 a;

  if(newsz < oldsz)
    return oldsz;

  oldsz = PGROUNDUP(oldsz);
  for(a = oldsz; a < newsz; a += PGSIZE){
    mem = kalloc();                    // 分配一页物理内存
    if(mem == 0){
      uvmdealloc(pagetable, a, oldsz);
      return 0;
    }
    memset(mem, 0, PGSIZE);            // 清零（防止信息泄露）
    if(mappages(pagetable, a, PGSIZE, (uint64)mem,
                PTE_W|PTE_X|PTE_R|PTE_U) != 0){
      kfree(mem);
      uvmdealloc(pagetable, a, oldsz);
      return 0;
    }
  }
  return newsz;
}
```

- 给进程的地址空间从 oldsz 增长到 newsz
- `PGROUNDUP(oldsz)` 向上对齐到页边界
- 逐页循环：`kalloc` 分配物理页 → `memset` 清零 → `mappages` 建立映射
- 权限 `PTE_W|PTE_X|PTE_R|PTE_U`：用户态可读写执行
- 这里是**非恒等映射**：虚拟地址 `a` 从 0 开始连续递增，物理地址 `mem` 是 `kalloc` 随机分配的
- 失败时回滚已分配的页（`uvmdealloc`）

`oldsz`、`newsz`、`a` 存的都是虚拟地址。在 xv6 中用户地址空间从 0 连续向上增长，所以"大小"和"顶部地址"是同一个数字。

### PGROUNDUP 宏

```c
#define PGROUNDUP(sz)  (((sz)+PGSIZE-1) & ~(PGSIZE-1))
```

把地址向上对齐到页边界（4096 的倍数）。先加 PGSIZE-1 再向下对齐，效果是：已经在边界上就不动，否则进到下一个页边界。

## 六、用户页表的释放

### 调用链与释放顺序

```
freeproc(p)
  ├── kfree(p->trapframe)                   释放 trapframe 物理页
  └── proc_freepagetable(p->pagetable, p->sz)
        ├── uvmunmap(TRAMPOLINE, 1, do_free=0)  清 PTE，不释放物理页（共享）
        ├── uvmunmap(TRAPFRAME,  1, do_free=0)  清 PTE，不释放物理页（已释放）
        └── uvmfree(pagetable, sz)
              ├── uvmunmap(0~sz, do_free=1)     清 PTE，释放用户数据物理页
              └── freewalk(pagetable)            递归释放所有页表页
```

顺序有严格约束：必须先清掉所有叶子 PTE，再调用 `freewalk` 释放页表页。`freewalk` 遇到有效叶子 PTE 会直接 panic，因为那意味着对应物理页将永远泄漏。

### freeproc

```c
static void freeproc(struct proc *p){
  if(p->trapframe)
    kfree((void*)p->trapframe);   // 单独释放 trapframe 物理页
  p->trapframe = 0;
  if(p->pagetable)
    proc_freepagetable(p->pagetable, p->sz);
  p->pagetable = 0;
  p->sz = 0;
  p->pid = 0;
  p->parent = 0;
  p->name[0] = 0;
  p->chan = 0;
  p->killed = 0;
  p->xstate = 0;
  p->state = UNUSED;
}
```

> [!TIP]
> **为什么 trapframe 要在 `freeproc` 里单独 `kfree`？**
>
> `uvmfree` 只释放虚拟地址 0 到 `p->sz` 范围内的页。trapframe 映射在 `TRAPFRAME`（`0x3ffffe000`），远在 `p->sz` 之上，`uvmfree` 根本不会遍历到它。
>
> `uvmunmap(pagetable, TRAPFRAME, 1, 0)` 最后一个参数是 0，只清 PTE，不释放物理页。所以必须在 `freeproc` 里提前 `kfree`，否则这页物理内存永远回收不了——PTE 清了，`p->trapframe` 也清零了，没有任何人还记得它的地址。

### proc_freepagetable

```c
void proc_freepagetable(pagetable_t pagetable, uint64 sz){
  uvmunmap(pagetable, TRAMPOLINE, 1, 0); // 清 PTE，不释放物理页（所有进程共享同一个 trampoline 物理页）
  uvmunmap(pagetable, TRAPFRAME,  1, 0); // 清 PTE，不释放物理页（已在 freeproc 中 kfree）
  uvmfree(pagetable, sz);
}
```

### uvmfree

```c
void uvmfree(pagetable_t pagetable, uint64 sz){
  if(sz > 0)
    uvmunmap(pagetable, 0, PGROUNDUP(sz)/PGSIZE, 1); // 清 PTE，并释放物理页
  freewalk(pagetable);  // 递归释放所有页表页
}
```

### uvmunmap

```c
void uvmunmap(pagetable_t pagetable, uint64 va, uint64 npages, int do_free){
  uint64 a;
  pte_t *pte;

  if((va % PGSIZE) != 0)
    panic("uvmunmap: not aligned");

  for(a = va; a < va + npages*PGSIZE; a += PGSIZE){
    if((pte = walk(pagetable, a, 0)) == 0)
      panic("uvmunmap: walk");
    if((*pte & PTE_V) == 0)
      panic("uvmunmap: not mapped");
    if(PTE_FLAGS(*pte) == PTE_V)
      panic("uvmunmap: not a leaf");
    if(do_free){
      uint64 pa = PTE2PA(*pte);
      kfree((void*)pa);           // 释放该 PTE 指向的物理页
    }
    *pte = 0;                     // 清除 PTE
  }
}
```

`do_free` 参数控制是否同时释放物理页：
- `do_free=1`：清 PTE + 释放物理页（用于释放用户数据页）
- `do_free=0`：只清 PTE（用于 TRAMPOLINE/TRAPFRAME，物理页另有处理）

### freewalk

```c
void freewalk(pagetable_t pagetable){
  for(int i = 0; i < 512; i++){
    pte_t pte = pagetable[i];
    if((pte & PTE_V) && (pte & (PTE_R|PTE_W|PTE_X)) == 0){
      // 非叶子 PTE：有效但无 R/W/X 权限位，说明指向下一级页表
      uint64 child = PTE2PA(pte);
      freewalk((pagetable_t)child);  // 递归释放子页表
      pagetable[i] = 0;
    } else if(pte & PTE_V){
      // 叶子 PTE 还有效，说明调用者忘了清理，物理页将泄漏
      panic("freewalk: leaf");
    }
  }
  kfree((void*)pagetable);  // 释放当前这一页页表页
}
```

叶子 PTE 和非叶子 PTE 的区分方式：有 R/W/X 权限位的是叶子（指向物理页），只有 V 位的是非叶子（指向下一级页表）。
