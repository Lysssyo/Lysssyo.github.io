# xv6 页表映射流程

以 `kvmmap(TRAMPOLINE, (uint64)trampoline, PGSIZE, PTE_R | PTE_X)` 为例，完整走一遍从虚拟地址到物理地址的映射建立过程。

## 背景

- `TRAMPOLINE` = `0x3ffffff000`，位于虚拟地址空间最顶端
- `trampoline` 的物理地址假设为 `0x80008000`（在内核代码段中）
- 这是一个**非恒等映射**：虚拟地址 ≠ 物理地址
- `kvminit()`方法里面调用`kvmmap`方法建立虚拟地址到物理地址的映射

## 调用链：kvmmap → mappages → walk

### 0. `kvminit`

```c
kernel_pagetable = (pagetable_t) kalloc(); 
memset(kernel_pagetable, 0, PGSIZE);
```

在调用kvmmap建立映射之前，根页表刚分配时被 `memset` 清零了，512 个 PTE 全是 0。

### 1. kvmmap —— 薄封装

```c
void kvmmap(uint64 va, uint64 pa, uint64 sz, int perm)
{
  if(mappages(kernel_pagetable, va, sz, pa, perm) != 0)
    panic("kvmmap");
}
```

自动传入 `kernel_pagetable`，失败则 panic。

### 2. mappages —— 逐页建立映射

`mappages`的任务是：**把一段连续的虚拟地址逐页映射到一段连续的物理地址。**

```c
int mappages(pagetable_t pagetable, uint64 va, uint64 size, uint64 pa, int perm)
{
  uint64 a, last;
  pte_t *pte;

  a = PGROUNDDOWN(va);
  last = PGROUNDDOWN(va + size - 1);
  for(;;){
    if((pte = walk(pagetable, a, 1)) == 0)
      return -1;
    if(*pte & PTE_V)
      panic("remap");
    *pte = PA2PTE(pa) | perm | PTE_V;
    if(a == last)
      break;
    a += PGSIZE;
    pa += PGSIZE;
  }
  return 0;
}
```

逐步看：

**先算边界**——`PGROUNDDOWN` 把起始和结束地址都对齐到页边界（4096 的倍数）。比如你要映射从 0x80001000 开始的 8192 字节，`a` 就是 0x80001000，`last` 就是 0x80002000。

**然后逐页循环**，每轮做三件事：

1. `walk(pagetable, a, 1)` — 拿着虚拟地址 `a` 去三级页表里找到对应的叶子 PTE 的**地址**。参数 `1` 表示"如果中间某级页表不存在就分配一个新的"。
2. `if(*pte & PTE_V) panic("remap")` — 检查这个 PTE 是不是已经有效了。如果是，说明这个虚拟地址已经被映射过了，重复映射是 bug，直接崩。
3. `*pte = PA2PTE(pa) | perm | PTE_V` — 这是核心的一行。把物理地址转成 PTE 格式（右移 12 位再左移 10 位，把物理页号放到 PTE 的正确位置），再加上权限位和 Valid 位，**写入 PTE**。

**最后**，`a` 和 `pa` 各自加一个 PGSIZE（4096），处理下一页，直到 `a == last` 为止。

本例只映射一页（PGSIZE = 4096），循环走一轮：

1. 调用 `walk(kernel_pagetable, 0x3ffffff000, 1)` 找到叶子 PTE 的地址
2. 检查 PTE 是否已有效（防止重复映射）
3. 写入 PTE：`*pte = PA2PTE(0x80008000) | PTE_R | PTE_X | PTE_V`

### 3. walk —— 三级页表遍历（核心）
![image.png](https://keith-knowledge-base.oss-accelerate.aliyuncs.com/20260409121620174.png)

```c
// shift a physical address to the right place for a PTE.
#define PA2PTE(pa) ((((uint64)pa) >> 12) << 10)

#define PTE2PA(pte) (((pte) >> 10) << 12)

pte_t *walk(pagetable_t pagetable, uint64 va, int alloc)
{
  if(va >= MAXVA)
    panic("walk");

  for(int level = 2; level > 0; level--) {
    pte_t *pte = &pagetable[PX(level, va)];
    if(*pte & PTE_V) {
      pagetable = (pagetable_t)PTE2PA(*pte);
    } else {
      if(!alloc || (pagetable = (pagetable_t)kalloc()) == 0)
        return 0;
      memset(pagetable, 0, PGSIZE);
      *pte = PA2PTE(pagetable) | PTE_V;//非叶子节点只有 PTE_V
    }
  }
  return &pagetable[PX(0, va)];
}
```

这个函数模拟的就是 RISC-V 硬件查页表的过程。传入一个虚拟地址，返回它对应的**最底层 PTE 的指针**。

**先理解 `PX` 宏**——它从虚拟地址中提取某一级的 9 位索引。Sv39 的虚拟地址 39 位中，高 27 位分成三组，每组 9 位：

- `PX(2, va)` 取最高 9 位（bit 30-38）→ 索引第 2 级（根页表）
- `PX(1, va)` 取中间 9 位（bit 21-29）→ 索引第 1 级
- `PX(0, va)` 取最低 9 位（bit 12-20）→ 索引第 0 级（叶子）

`0x3ffffff000` 的低 39 位：

```
0b 111111111     111111111     111111111    000000000000
   PX(2,va)=511  PX(1,va)=511  PX(0,va)=511  offset=0
```

> [!TIP]  `PX` 宏的计算方式：
> 
> ```c
> #define PX(level, va) ((((uint64)(va)) >> (12 + 9*(level))) & 0x1FF)
> ```
> 
> 以 PX(2) 为例：右移 30 位，取低 9 位，得到 511。

#### 逐级遍历

**level=2（根页表）：** `&pagetable[511]` → 根页表第 511 号 PTE。无效 → `kalloc()` 分配新页（假设 `0x87f30000`），清零，将其物理地址写入该 PTE。`pagetable` 更新为 `0x87f30000`。

**level=1（二级页表）：** `&pagetable[511]` → `0x87f30000` 页的第 511 号 PTE。无效 → `kalloc()` 分配新页（假设 `0x87f31000`），清零，将其物理地址写入该 PTE。`pagetable` 更新为 `0x87f31000`。

**循环结束（三级页表）：** 返回 `&pagetable[511]`，即 `0x87f31000` 页中第 511 号 PTE 的地址。

### 4. 写入 PTE

回到 mappages，往 walk 返回的 PTE 地址写入：

```c
*pte = PA2PTE(0x80008000) | PTE_R | PTE_X | PTE_V;
```

PTE 的 64 位布局：

```
bit 63..54:  保留
bit 53..10:  物理页号 (PPN) = 0x80008000 >> 12 = 0x80008
bit 9..0:    标志位 (R=1, X=1, V=1)
```

写 PTE 的本质：往一个 `uint64` 里写一个按硬件规定格式编码的数字。

## 写入 vs 读取

**写入（软件，启动时）：** `kvmmap` → `mappages` → `walk`，在页表的对应位置写好 PTE，为子页表分配内存。

**读取（硬件，运行时）：** CPU 访问虚拟地址 `0x3ffffff000` 时，硬件自动做与 `walk` 相同的事——从 `satp` 指向的根页表开始，逐级查找：根页表`[511]` → 二级页表`[511]` → 三级页表`[511]`，取出物理页号，得到物理地址 `0x80008000`。

## 恒等映射 vs 非恒等映射

恒等映射（内核大部分区域）：`kvmmap(KERNBASE, KERNBASE, ...)`，虚拟地址 = 物理地址，方便内核代码直接使用物理地址。

非恒等映射（如 TRAMPOLINE）：`kvmmap(TRAMPOLINE, (uint64)trampoline, ...)`，虚拟地址 ≠ 物理地址，页表负责连接两者。

恒等映射是设计选择，不是硬件要求。用户进程的页表完全不是恒等映射：每个进程以为自己从虚拟地址 0 开始，实际物理页散布在内存各处。
