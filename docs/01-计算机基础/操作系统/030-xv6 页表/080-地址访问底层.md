# xv6 地址访问底层

## 一、C 代码里的指针都是虚拟地址

**C 语言本身没有"虚拟地址"和"物理地址"的区分，编译器只知道指针是一个数字**。但在开启分页的系统中，**所有指针操作用的都是虚拟地址**，由 MMU 硬件通过页表翻译成物理地址。

```c
char *p = (char *)0x1000;
char c = *p;
```

`(char *)0x1000` 把数字 0x1000 强转成指针。`*p` 读取该地址的值。底层发生的事：

```
1. 编译器生成加载指令：lb a0, 0(a1)   // a1 = 0x1000
2. CPU 执行这条指令：
   ① 取出虚拟地址 0x1000
   ② 查 TLB，看有没有缓存
   ③ 未命中则走三级页表翻译：
      PX(2) → 根页表找到二级页表地址
      PX(1) → 二级页表找到三级页表地址
      PX(0) → 三级页表找到物理页号
   ④ 物理页号 + 页内偏移 → 物理地址
   ⑤ 去物理内存读取数据
   ⑥ 把值放到寄存器
```

整个翻译过程对 C 代码完全透明，程序员不需要知道物理地址是什么。

## 二、地址翻译是 MMU 硬件做的

**地址翻译由 CPU 内部的 MMU（Memory Management Unit）硬件自动完成**，不是软件做的。`satp` 寄存器控制 MMU 的行为：

- `satp = 0`（分页未开启）：CPU 执行 `*p` → 直接访问物理地址，不做翻译
- `satp ≠ 0`（分页已开启）：CPU 执行 `*p` → MMU 自动查页表翻译 → 物理地址

同样的 C 代码、同样的机器指令，`satp` 设置前后行为完全不同。


## 三、内核启动：从无分页到有分页

内核启动的 `main()` 一开始没有开启分页，CPU 直接用物理地址访问内存：

```c
// main.c
void main() {
  kinit();           // 初始化物理内存分配器（此时无分页，地址就是物理地址）
  kvminit();         // 建立内核页表（只是填写页表数据结构，还没开启）
  kvminithart();     // 写 satp 寄存器，开启分页
  // ... 后续代码在分页模式下运行
}
```

`kvminithart()` ：

```c
void kvminithart() {
  w_satp(MAKE_SATP(kernel_pagetable));  // 开启分页
  sfence_vma();                          // 刷新 TLB
}
```

**执行 `w_satp` 之后，CPU 立刻从"直接访问物理地址"切换到"所有地址经 MMU 翻译"**。**为什么没有崩？** 因为内核页表是恒等映射——虚拟地址 = 物理地址。切换前 PC 指向物理地址 `0x800xxxxx`，切换后 MMU 把虚拟地址 `0x800xxxxx` 翻译成物理地址 `0x800xxxxx`，结果一样，代码无缝继续执行。

## 四、kalloc() 返回的是指针

`kalloc()` 返回类型是 `void *`（指针），但数值上等于物理地址。

因为内核是恒等映射，同一个数字同时是合法的虚拟地址和物理地址，所以可以一值两用：

```c
char *mem = kalloc();            // 返回物理地址 0x87f50000
memset(mem, 0, PGSIZE);          // 当指针用 ✓（CPU 用虚拟地址 0x87f50000，恒等映射翻译后还是 0x87f50000）
mappages(pt, va, PGSIZE, (uint64)mem, ...);  // 当物理地址用 ✓（编码进 PTE 的物理页号字段）
```

如果内核不是恒等映射，物理地址就不能直接当指针用，需要先建映射拿到虚拟地址。Linux 里就有 `phys_to_virt()` 函数做转换。xv6 用恒等映射省去了这个麻烦。

Linux 的 `kmalloc`（相当于 xv6 的 `kalloc`）返回的是**虚拟地址**，是内核虚拟映射中的地址，这块区域在物理内存中也是连续的。

Linux 内核不是恒等映射，而是有一个固定偏移。比如在 32 位 Linux 上，物理地址 `0x00000000` 映射到虚拟地址 `0xC0000000`，物理地址 `0x00100000` 映射到虚拟地址 `0xC0100000`。虚拟地址 = 物理地址 + `PAGE_OFFSET`。

所以如果需要物理地址（比如给 DMA 硬件用，或者填页表），必须做转换：

```c
void *vaddr = kmalloc(4096, GFP_KERNEL);   // 返回虚拟地址，如 0xC0500000
unsigned long paddr = virt_to_phys(vaddr);  // 转成物理地址，如 0x00500000
```

对比 xv6 和 Linux：

```
xv6:
  kalloc() 返回 0x87f50000
  当指针用：直接用 0x87f50000（恒等映射，虚拟 = 物理）
  当物理地址：直接用 0x87f50000（数值相同）

Linux:
  kmalloc() 返回 0xC0500000（虚拟地址）
  当指针用：直接用 0xC0500000（内核页表有映射）
  当物理地址：必须 virt_to_phys() 转换成 0x00500000
```

## 五、用户态只能拿到虚拟地址

用户程序不能调用 `kalloc`（它是内核函数，在 `kernel/kalloc.c` 里）。用户程序要分配内存，只能通过系统调用 `sbrk()`，拿到的是**虚拟地址**，永远不知道背后的物理地址。

假设当前 `p->sz = 0x3000`（关于什么是 `p->sz` 见[用户虚拟地址空间布局](050-xv6%20用户页表.md#二、用户虚拟地址空间布局)），用户调用 `sbrk(4096)`：

```
用户程序：
  addr = sbrk(4096);   // 返回 0x3000（旧的 p->sz）

内核内部发生的事：
  uvmalloc(pagetable, 0x3000, 0x4000)
    → mem = kalloc()       // 分配物理页，返回物理地址 0x87f50000
    → memset(mem, 0, PGSIZE)
    → mappages(pagetable, 0x3000, PGSIZE, 0x87f50000, PTE_W|PTE_R|PTE_U)
      // 在用户页表中建立映射：虚拟地址 0x3000 → 物理地址 0x87f50000
    → 返回 newsz = 0x4000

  p->sz = 0x4000    // 更新进程大小
  返回 0x3000 给用户  // 返回旧 sz，即新分配区域的起始虚拟地址
```

用户程序拿到 `0x3000`，直接当指针用：

```c
char *p = sbrk(4096);    // p = 0x3000
*p = 'A';                // 写虚拟地址 0x3000
                         // CPU 中的 MMU 查用户页表 → 物理地址 0x87f50000
                         // 写入物理内存
```

`sbrk` 返回的不是 `kalloc` 的返回值，而是 `p->sz`（虚拟地址空间的边界）。用户程序从头到尾只看到虚拟地址 `0x3000`，不知道物理地址 `0x87f50000` 的存在。

同一个物理页，**内核态用 `0x87f50000` 访问（恒等映射），用户态用 `0x3000` 访问（用户页表映射）**。数值不同，但到达同一物理位置。
