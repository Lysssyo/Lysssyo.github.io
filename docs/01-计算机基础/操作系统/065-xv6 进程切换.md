# xv6 进程切换

## 一、概述

从进程 A 切换到进程 B，需要跨越两道边界：

```
用户态 ──────── 边界1：用户态/内核态（trampoline）──────── 内核态
进程 A ──────── 边界2：进程切换（swtch）─────────────── 进程 B
```

- **边界 1**：同一个进程在用户态和内核态之间切换，通过 trampoline 完成，**切换页表 + 保存全部寄存器**
- **边界 2**：CPU 从一个进程切换到另一个进程，通过 `swtch` 完成，**只切换 callee-saved 寄存器，不切换页表**

两个机制完全独立，组合起来完成 A 用户态 → B 用户态的完整切换。

## 二、关键数据结构

理解进程切换需要先认识三个数据结构，它们各自服务于不同的机制：

### trapframe — 跨越边界 1 用

```c
struct trapframe {
  /*   0 */ uint64 kernel_satp;   // 内核页表地址，uservec 用它切换 satp
  /*   8 */ uint64 kernel_sp;     // 该进程的内核栈顶，uservec 用它设置 sp
  /*  16 */ uint64 kernel_trap;   // usertrap() 的地址，uservec 最后跳转到这里
  /*  24 */ uint64 epc;           // 用户态程序计数器，sret 返回时跳到这里
  /*  32 */ uint64 kernel_hartid; // CPU 编号，uservec 用它设置 tp
  /*  40 */ uint64 ra;            // 保存用户态程序的返回地址，trap 返回后用户代码继续执行函数调用链
  /*  48 */ uint64 sp;            // 用户栈指针
  /*  56 */ uint64 gp;
  /*  64 */ uint64 tp;
  /*  72 */ uint64 t0;
  /*  80 */ uint64 t1;
  /*  88 */ uint64 t2;
  /*  96 */ uint64 s0;
  /* 104 */ uint64 s1;
  /* 112 */ uint64 a0;            // 系统调用参数 / 返回值
  /* 120 */ uint64 a1;            // 系统调用参数
  /* 128 */ uint64 a2;
  /* 136 */ uint64 a3;
  /* 144 */ uint64 a4;
  /* 152 */ uint64 a5;
  /* 160 */ uint64 a6;
  /* 168 */ uint64 a7;            // 系统调用号
  /* 176 */ uint64 s2;
  ...                             // s3-s11, t3-t6
  /* 280 */ uint64 t6;
};
```

每个进程一个，在 `allocproc` 中 `kalloc` 分配。trampoline 代码在进入/离开内核时，通过它保存和恢复**全部 32 个用户寄存器**。

详见 [xv6 trampoline 与 trapframe](050-%20xv6%20trampoline%20与%20trapframe.md) 以及 [allocproc — 进程的骨架](060-xv6%20进程创建流程.md#二、allocproc%20—%20进程的骨架)。

### context — 跨越边界 2 用

```c
struct context {
  uint64 ra;   // 内核态的返回地址，swtch 的 ret 跳到这里
  uint64 sp;   // 内核栈指针
  uint64 s0;
  uint64 s1;
  // ... s2-s11
};
```


每个进程一个（`p->context`），每个 CPU 一个（`c->context`，给 scheduler 用）。`swtch` 通过它保存和恢复 **14 个 callee-saved 寄存器（被调用者保存寄存器，由”被调用者”负责保存）**。

> [!TIP]
> **trapframe 和 context 都保存了 ra 和 sp，但属于不同层次：**
>
> | | trapframe（边界1） | context（边界2） |
> |---|---|---|
> | **ra** | 用户态返回地址（如 `foo` 调用 `bar` 的下一行） | 内核态返回地址（`sched` 中 `swtch` 的下一行） |
> | **sp** | 用户栈指针 | 内核栈指针 |
> | **保存时机** | trap 发生时，`uservec` 保存 | 进程让出 CPU 时，`swtch` 保存 |
> | **恢复时机** | 返回用户态时，`userret` 恢复 | 被调度回来时，`swtch` 恢复 |
>
> 切回进程 A 时，先从 context 恢复内核态的 ra/sp（回到 `sched`），一路返回到 `usertrapret`；再从 trapframe 恢复用户态的 ra/sp（回到用户代码）。两层嵌套，各管各的。

> [!TIP]
> **为什么 trapframe 保存全部 32 个寄存器，context 只保存 14 个？**
>
> trapframe（边界1）：trap 是**硬件强行打断**用户程序，用户代码完全不知情，没有任何机会保存自己的寄存器，所以 `uservec` 必须保存全部 32 个，一个都不能丢。
>
> context（边界2）：`swtch` 是被 C 函数 `sched()` **正常调用**的。编译器在生成调用代码时，已经按 calling convention 把 caller-saved 寄存器（a0-a7、t0-t6）保存到内核栈上了。`swtch` 只需要保存编译器管不到的 callee-saved 部分（ra、sp、s0-s11）。切回来时，`swtch` 恢复一半，编译器从栈上恢复另一半，合起来所有寄存器都恢复了。

### 内核栈 — 内核态执行用

每个进程一个，一页大小（4096 字节），在 `procinit` 中分配。进程在内核态执行时的函数调用栈。进程被切走时，栈上的内容保留（调用链、局部变量等），切回来后继续使用。

详见 [内核栈的设置：procinit](055-xv6%20内核页表.md#五、内核栈的设置：procinit)

## 三、swtch：进程切换的核心

`swtch` 是整个进程切换的核心，一段很短的汇编：

```asm
swtch:
        # a0 = 要保存到的 context 地址
        # a1 = 要恢复的 context 地址

        # 保存当前寄存器到 a0 指向的 context
        sd ra, 0(a0)
        sd sp, 8(a0)
        sd s0, 16(a0)
        sd s1, 24(a0)
        ... # s2-s11

        # 从 a1 指向的 context 恢复寄存器
        ld ra, 0(a1)
        ld sp, 8(a1)
        ld s0, 16(a1)
        ld s1, 24(a1)
        ... # s2-s11

        ret    # 跳到刚恢复的 ra
```

> [!TIP]
> `sd` = store double（8 字节从寄存器存到内存），`ld` = load double（8 字节从内存读到寄存器），`ret` = 跳转到 `ra` 寄存器的地址执行（等价于 `jalr x0, ra, 0`）。

`swtch` 做的事很简单：把当前的 ra、sp、s0-s11 存到一个 context，再从另一个 context 恢复 ra、sp、s0-s11，然后 `ret`。但效果很深刻：

- **`ra` 被换了** → `ret` 跳到的地方变了（从"我上次在哪"变成"对方上次在哪"）
- **`sp` 被换了** → 函数栈变了（从我的内核栈变成对方的内核栈）
- **不切换页表** → 全程在内核页表下执行，satp 不变

`swtch` 总是在**进程和 scheduler 之间**成对调用：

```
进程 → scheduler：swtch(&p->context, &c->context)
scheduler → 进程：swtch(&c->context, &p->context)
```

从不直接从进程 A 切到进程 B，中间一定经过 scheduler。

## 四、scheduler：调度的中枢

```c
void scheduler(void){
  struct proc *p;
  struct cpu *c = mycpu();

  c->proc = 0;
  for(;;){
    intr_on();

    int found = 0;
    for(p = proc; p < &proc[NPROC]; p++) {
      acquire(&p->lock);
      if(p->state == RUNNABLE) {
        p->state = RUNNING;
        c->proc = p;
        swtch(&c->context, &p->context);  // ← 切到进程，进程让出后从这里返回

        c->proc = 0;
        found = 1;
      }
      release(&p->lock);
    }

    if(found == 0) {
      intr_on();
      asm volatile("wfi");    // 没有可运行进程，等待中断
    }
  }
}
```

scheduler **不是进程**。它是每个 CPU 核心的一段固定代码，没有 PID，不在进程表里。它有自己的 context 和栈，只是为了让 `swtch` 有地方保存和恢复寄存器。可以把它理解为 CPU 的"待机循环"。

scheduler 的生命就是一个无限循环：遍历进程表 → 找到 RUNNABLE 的进程 → `swtch` 切过去 → 进程让出 CPU 后回到 `swtch` 的下一行 → 继续遍历。

## 五、完整流程：从进程 A 到进程 B

### 总览

```
进程 A 用户态   satp = A 的用户页表
  │ trap（时钟中断）
  │ trampoline: uservec（最后调用usertrap()） ← 边界 1：进入内核
  ↓
进程 A 内核态   satp = 全局内核页表
  │ usertrap() → yield() → sched()
  │ swtch(&p->context, &c->context)   ← 边界 2：A 切到 scheduler
  ↓
scheduler       satp = 全局内核页表（不变）
  │ 找到进程 B
  │ swtch(&c->context, &p->context)   ← 边界 2：scheduler 切到 B
  ↓
进程 B 内核态   satp = 全局内核页表（不变）
  │ sched() 返回 → usertrapret()
  │ trampoline: userret          ← 边界 1：返回用户态
  ↓
进程 B 用户态   satp = B 的用户页表
```

下面展开每一步。

### 5.1 trap 触发：A 用户态 → A 内核态

进程 A 在用户态运行，时钟中断触发 trap。**CPU 硬件自动**：

1. 把当前 PC 保存到 `sepc`
2. 把原因保存到 `scause`
3. 切换到 supervisor mode
4. 跳转到 `stvec` 指向的地址（`uservec`）

> [!TIP]
> Trap 是 CPU 从用户态切换到内核态的统称，有三种：**系统调用**（`ecall` 指令）、**异常**（page fault、非法指令）、**中断**（时钟、磁盘 I/O）。

然后 trampoline 的 `uservec` 执行：保存全部寄存器到 trapframe → 切换 satp 到内核页表 → 跳转到 `usertrap()`。

具体参考：[trampoline 代码的工作流程](050-%20xv6%20trampoline%20与%20trapframe.md#五、trampoline%20代码的工作流程)

此时**进程 A 已进入内核态**，satp = 全局内核页表，sp = 内核栈指针。

### 5.2 usertrap：判断 trap 类型并处理

```c
void usertrap(void){
  // ...

  if(r_scause() == 8){
    // 系统调用
    p->trapframe->epc += 4;
    intr_on();
    syscall();
  } else if((which_dev = devintr()) != 0){
    // 设备中断
  } else {
    // 异常
    p->killed = 1;
  }

  // 时钟中断 → 让出 CPU → 触发进程切换
  if(which_dev == 2)
    yield();

  usertrapret();
}
```

对于进程切换来说，关键是最后几行：如果是时钟中断（`which_dev == 2`），调用 `yield()`。这就是从"trap 处理"接到"进程切换"的桥梁。

### 5.3 yield → sched → swtch：A 切到 scheduler

```c
void yield(void) {
  struct proc *p = myproc();
  acquire(&p->lock);
  p->state = RUNNABLE;     // A 标记为"可运行但不在运行"
  sched();                  // 进入 sched
  release(&p->lock);        // ← A 被切回来后，从这里继续
}

void sched(void) {
  struct proc *p = myproc();
  swtch(&p->context, &mycpu()->context);
  mycpu()->intena = intena; // ← A 被切回来后，从这里继续（返回到 yield 的 release）
}
```

`swtch` 保存 A 的寄存器到 `p->context`，恢复 scheduler 的寄存器。`ret` 跳到 scheduler 上次调用 `swtch` 时保存的 `ra`——也就是 scheduler 循环中 `swtch(&c->context, &p->context)` 的下一行。

此时 CPU 从"在 A 的内核栈上执行 sched"变成"在 scheduler 的栈上执行 scheduler 循环"。**页表没有变**，全程是全局内核页表。

### 5.4 scheduler 选中 B

scheduler 回到循环，继续遍历进程表，找到下一个 `state == RUNNABLE` 的进程 B：

```c
p->state = RUNNING;
c->proc = p;
swtch(&c->context, &p->context);  // 保存 scheduler 寄存器，恢复 B 的寄存器
```

`swtch` 恢复 B 的 `context`。`ret` 跳到 B 上次在 `sched()` 里调用 `swtch` 时保存的 `ra`——也就是 `sched()` 中 `swtch` 的下一行。

CPU 从 scheduler 的栈切到 B 的内核栈，B 从上次被切走的位置继续执行。

### 5.5 B 内核态 → B 用户态

B 从 `sched()` 的 `swtch` 下一行继续，一路返回：

```
sched() 返回 → yield() 中的 release(&p->lock) → yield() 返回
→ usertrap() 末尾的 usertrapret()
→ usertrapret() 跳到 trampoline 的 userret
→ userret 切换 satp 到 B 的用户页表，恢复寄存器
→ sret 回到用户态，PC 跳到 sepc（B 上次被中断的位置）
```

`usertrapret()` 准备好 trapframe 信息后跳到 `userret`：

```c
void usertrapret(void){
  // ...
  // set up trapframe values that uservec will need when
  // the process next re-enters the kernel.
  p->trapframe->kernel_satp = r_satp();         // kernel page table
  p->trapframe->kernel_sp = p->kstack + PGSIZE; // process's kernel stack
  p->trapframe->kernel_trap = (uint64)usertrap;
  p->trapframe->kernel_hartid = r_tp();         // hartid for cpuid()
  
  //...

  uint64 fn = TRAMPOLINE + (userret - trampoline);
  ((void (*)(uint64, uint64))fn)(TRAPFRAME, satp);
}
```

```asm
userret:
        csrw satp, a1    # 切换到 B 的用户页表
        sfence.vma

        ld ra, 40(a0)    # 从 trapframe 恢复所有寄存器
        ld sp, 48(a0)
        ...

        sret             # 切回 user mode，跳到 sepc
```

至此，CPU 从运行进程 A 切换到了运行进程 B。

## 六、新进程第一次被调度

新进程从未在用户态运行过，没有经历过"用户态 → trap → 内核态"。但返回用户态的唯一路径是 `usertrapret` → `userret` → `sret`。怎么办？

**假装从 trap 返回**：`allocproc` 手动设置 context，让新进程第一次被 `swtch` 调度时走一条特殊路径进入用户态。

### allocproc 中的设置

```c
memset(&p->context, 0, sizeof(p->context));
p->context.ra = (uint64)forkret;       // 不是 sched 的返回地址，而是 forkret
p->context.sp = p->kstack + PGSIZE;    // 内核栈底
```

### 调度路径

```
scheduler
  → swtch(&c->context, &p->context)
  → ret 跳到 ra = forkret（不是 sched 的返回处）
  → forkret() 调用 usertrapret()
  → usertrapret() 跳到 userret
  → userret: sret 进入用户态
  → PC 跳到 trapframe->epc
```

对于 `userinit` 创建的第一个进程，`epc = 0`，从虚拟地址 0 开始执行 initcode。

### 与正常切换的对比

```
正常进程（被切走再切回）：
  swtch → ret → sched() 返回 → yield() 返回 → usertrap() → usertrapret()

新进程（第一次被调度）：
  swtch → ret → forkret() → usertrapret()
```

区别只在 `ret` 跳到哪：正常进程跳回 `sched()` 中 `swtch` 的下一行（上次被切走时保存的 `ra`），新进程跳到 `forkret`（`allocproc` 手动设置的 `ra`）。之后都汇入 `usertrapret` → `userret` → 用户态。
