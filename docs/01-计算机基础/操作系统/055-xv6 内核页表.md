# xv6 内核页表

## 一、内核页表概述

xv6 有一个全局内核页表 `kernel_pagetable`，所有进程在内核态共享这一张页表。内核页表的核心特点是**恒等映射： 虚拟地址 = 物理地址（trampoline 和内核栈除外）**。

恒等映射的好处：内核代码里不需要区分虚拟地址和物理地址，`kalloc()` 返回的物理地址可以直接当指针用。`walk` 函数从 PTE 中取出下一级页表的物理地址，也能直接当虚拟地址解引用，因为两者相等。

## 二、内核虚拟地址空间布局

```
0x3ffffff000 ─── TRAMPOLINE（非恒等映射，显式映射到 trampoline 代码的物理页）
                 kstack 0（进程 0 的内核栈）
                 guard page（无映射）
                 kstack 1（进程 1 的内核栈）
                 guard page
                 ...（64 个进程的内核栈 + guard page）
                 （中间大片未映射区域）
PHYSTOP ──────── 恒等映射到此为止
                 空闲物理内存（free memory）
   etext ─────── 内核数据段起始（可读可写）
KERNBASE ─────── 内核代码段起始（可读可执行）
                 PLIC（恒等映射）
                 CLINT（恒等映射）
                 VIRTIO（恒等映射）
                 UART（恒等映射）
0x0 ──────────── 最低地址
```

TRAMPOLINE 的虚拟地址（`0x3ffffff000`）远超物理内存范围，不在恒等映射覆盖的区域内，所以必须显式映射。内核栈使用 `KSTACK` 宏计算虚拟地址，也不是恒等映射。

## 三、内核页表的创建：kvminit

```c
void kvminit(){
  kernel_pagetable = (pagetable_t) kalloc();
  memset(kernel_pagetable, 0, PGSIZE);

  kvmmap(UART0, UART0, PGSIZE, PTE_R | PTE_W);           // MMIO 设备
  kvmmap(VIRTIO0, VIRTIO0, PGSIZE, PTE_R | PTE_W);       // MMIO 设备
  kvmmap(CLINT, CLINT, 0x10000, PTE_R | PTE_W);           // MMIO 设备
  kvmmap(PLIC, PLIC, 0x400000, PTE_R | PTE_W);            // MMIO 设备
  kvmmap(KERNBASE, KERNBASE, (uint64)etext-KERNBASE, PTE_R | PTE_X);  // 内核代码段
  kvmmap((uint64)etext, (uint64)etext, PHYSTOP-(uint64)etext, PTE_R | PTE_W);  // 内核数据段+空闲内存
  kvmmap(TRAMPOLINE, (uint64)trampoline, PGSIZE, PTE_R | PTE_X);  // trampoline
}
```

关于 `kvmmap` 是怎么建立映射的，参考[040-xv6 页表映射流程](040-xv6%20页表映射流程.md)

要点：

- 此时分页还没开启，CPU 直接用物理地址访问内存
- 前两个参数几乎都一样（恒等映射），除了 TRAMPOLINE
- `etext` 是链接器设置的符号，标记内核代码段的结束位置
- 代码段权限 `PTE_R | PTE_X`（不可写），数据段权限 `PTE_R | PTE_W`（不可执行）
- 整个函数只是在内存里建好页表数据结构，等 `kvminithart` 写 `satp` 后才生效

## 四、内核页表的安装：kvminithart

```c
void kvminithart(){
  w_satp(MAKE_SATP(kernel_pagetable));
  sfence_vma();
}
```

`MAKE_SATP` 宏把页表物理地址转换成 `satp` 寄存器需要的格式：

```c
#define MAKE_SATP(pagetable) (SATP_SV39 | (((uint64)pagetable) >> 12))

// use riscv's sv39 page table scheme.
#define SATP_SV39 (8L << 60)
```

- `>> 12`：物理地址转物理页号（PPN）
- `SATP_SV39`：在高位设置模式位，告诉 CPU 使用 Sv39 分页

`satp` 寄存器的布局：

```
bit 63-60:  模式（8 = Sv39）
bit 59-44:  ASID（xv6 不用，全 0）
bit 43-0:   根页表的物理页号（PPN）
```

写入 `satp` 后，CPU 开启分页。因为是恒等映射，下一条指令的虚拟地址等于物理地址，执行不会断裂。`sfence_vma()` 刷新 TLB，确保旧缓存被清掉。

> [!TIP]
> TLB（Translation Lookaside Buffer）是 CPU 里的硬件缓存，缓存"虚拟页号 → 物理页号"的翻译结果。
> 
> 正常三级页表翻译需要三次内存访问加一次目标访问（共四次），TLB 命中时只需一次。
> 
> 切换页表后必须刷新 TLB（`sfence.vma` 指令），否则 CPU 可能用旧映射访问错误的物理页。xv6 在两个地方执行 `sfence.vma`：
> 
> - `kvminithart` 中重载 satp 后
> - trampoline 中切换用户/内核页表后

## 五、内核栈的设置：procinit

```c
void procinit(void){
  struct proc *p;
  initlock(&pid_lock, "nextpid");
  for(p = proc; p < &proc[NPROC]; p++) {
      initlock(&p->lock, "proc");
      char *pa = kalloc();
      if(pa == 0) panic("kalloc");
      uint64 va = KSTACK((int) (p - proc));
      kvmmap(va, (uint64)pa, PGSIZE, PTE_R | PTE_W);
      p->kstack = va; // 存的是栈的起始虚拟地址，也是栈的低地址端，栈从高地址往低地址生长
  }
  kvminithart();
}
```

- 内核启动时为所有 64 个进程预先分配内核栈
- `KSTACK(p)` 宏：`(TRAMPOLINE - ((p)+1) * 2 * PGSIZE)`，从 TRAMPOLINE 往下排列
- 乘以 2 是因为每个进程占两页：一页是栈，一页是 guard page（无映射）
- 内核栈权限 `PTE_R | PTE_W`，没有 `PTE_X`（栈上的数据不应该被执行）
- guard page 没有映射，栈溢出时硬件触发 page fault
- 最后调用 `kvminithart()` 重载 satp，让硬件知道新的 PTE

> [!TIP]
> 以上三个方法（`kvminit`,`kvminithart`,`procinit`）都是在启动内核的 `main` 方法调用。

## 六、内核栈的作用

进程在内核态执行时需要栈（跟用户态需要用户栈一样），用来存函数调用的局部变量、返回地址等。

每个进程需要自己的内核栈，因为：

- 进入内核态后不能用用户栈（安全隔离，且切换页表后用户栈可能不可访问）
- 多个进程可能同时处于内核态（多核 CPU），共享栈会互相覆盖

内核栈大小固定一页（4096 字节），不会自动增长。内核代码不应在栈上分配大数组或写深递归，需要大块内存应用 `kalloc()`。

trap 进入内核时，trampoline 从 trapframe 加载内核栈指针：

```asm
ld sp, 8(a0)    # 从 trapframe 取出内核栈指针
```

## 七、恒等映射 vs 非恒等映射

**恒等映射**（内核大部分区域）：`kvmmap(KERNBASE, KERNBASE, ...)`，虚拟地址 = 物理地址。方便内核代码直接使用物理地址作为指针。

**非恒等映射**（trampoline、内核栈）：虚拟地址 ≠ 物理地址，页表负责连接两者。

恒等映射是**设计选择**，不是硬件要求。用户进程的页表完全不是恒等映射：每个进程以为自己从虚拟地址 0 开始，实际物理页散布在内存各处。

## 八、启动流程中的页表时间线

```
1. kvminit()      — 在内存中建好内核页表数据结构（此时分页未开启）
2. kvminithart()  — 写 satp，开启分页，刷新 TLB
3. procinit()     — 分配内核栈，映射到内核页表，重载 satp
```

kvminit 只被 CPU 0 调用一次（在 `main` 中）。其他 CPU 等 CPU 0 初始化完成后，各自调用 `kvminithart()` 安装同一个内核页表到自己的 satp。最终所有 CPU 都进入 `scheduler()`。
