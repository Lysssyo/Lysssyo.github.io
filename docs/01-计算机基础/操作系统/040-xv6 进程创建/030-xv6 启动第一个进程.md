# xv6 启动与第一个进程

## 一、启动流程概述

机器上电后，经历三个阶段：汇编入口 → C 语言初始化 → 所有 CPU 进入 scheduler。

```
_entry（汇编）→ start() → main() → scheduler()
```

## 二、`_entry`：所有 CPU 的入口

```asm
_entry:
        la sp, stack0
        li a0, 1024*4
        csrr a1, mhartid    # 读取 CPU 编号
        addi a1, a1, 1
        mul a0, a0, a1
        add sp, sp, a0      # 每个 CPU 设置自己的栈

        call start           # 所有 CPU 都跳到 start
```

每个 CPU 都执行这段代码，各自设置独立的栈后跳到 `start()`。

## 三、main：只有 CPU 0 做全局初始化

所有 CPU 都会进入 `main()`，但只有 CPU 0 做初始化：

```c
void main(){
  if(cpuid() == 0){
    // === CPU 0：全局初始化 ===
    consoleinit();          // 控制台
    printfinit();           // printf
    kinit();                // 物理内存分配器
    kvminit();              // 创建内核页表（此时分页未开启）
    kvminithart();          // 写 satp，开启分页
    procinit();             // 初始化锁，分配内核栈
    trapinit();             // trap 初始化
    trapinithart();         // 设置 stvec
    plicinit();             // 中断控制器
    plicinithart();
    binit();                // 块缓存
    iinit();                // inode 缓存
    fileinit();             // 文件表
    virtio_disk_init();     // 磁盘
    userinit();             // 创建第一个进程
    __sync_synchronize();   // 内存屏障
    started = 1;            // 告诉其他 CPU：初始化完成

  } else {
    // === 其他 CPU：等待 CPU 0 完成 ===
    while(started == 0)
      ;                     // 空转等待
    __sync_synchronize();
    kvminithart();          // 安装内核页表到自己的 satp
    trapinithart();
    plicinithart();
  }

  scheduler();              // 所有 CPU 最终都进入 scheduler
}
```

### 关键时间线

1. `kvminit()`：在内存中建好内核页表数据结构（分页未开启，地址直接是物理地址）
2. `kvminithart()`：写 `satp` 开启分页（恒等映射，下一条指令不断裂）
3. `procinit()`：初始化锁，为 64 个进程分配内核栈
4. `userinit()`：创建第一个进程
5. `scheduler()`：进入调度循环，永不返回

## 四、userinit：创建第一个进程

```c
void userinit(void){
  struct proc *p;

  p = allocproc();          // 分配 PID、trapframe、用户页表、内核上下文

  // 把 initcode 的机器码复制到进程的用户内存
  uvminit(p->pagetable, initcode, sizeof(initcode));
  p->sz = PGSIZE;           // 用户内存大小 = 一页

  // 设置 trapframe，让进程从地址 0 开始执行
  p->trapframe->epc = 0;    // 程序计数器：从虚拟地址 0 开始
  p->trapframe->sp = PGSIZE; // 栈指针：页顶（向下生长）

  safestrcpy(p->name, "initcode", sizeof(p->name));
  p->state = RUNNABLE;       // 标记为可运行
}
```

### initcode 是什么

`initcode` 是一小段预编译好的汇编机器码，硬编码在 `kernel/proc.c` 里：

```c
uchar initcode[] = {
  0x17, 0x05, 0x00, 0x00, 0x13, 0x05, ...
};
```

这段代码做的事只有一件：执行 `exec("/init", argv)`。

### uvminit 做了什么

`kalloc` 分配一页物理内存 → 把 initcode 的内容复制进去 → `mappages` 映射到虚拟地址 0。这样进程从虚拟地址 0 开始执行时就能跑到 initcode。

## 五、scheduler：调度第一个进程

```c
void scheduler(void){
  struct proc *p;
  struct cpu *c = mycpu();

  c->proc = 0;
  for(;;){
    intr_on();              // 开启中断

    int found = 0;
    for(p = proc; p < &proc[NPROC]; p++) {
      acquire(&p->lock);
      if(p->state == RUNNABLE) {
        p->state = RUNNING;
        c->proc = p;
        swtch(&c->context, &p->context);  // 切换到进程
        c->proc = 0;
        found = 1;
      }
      release(&p->lock);
    }

    if(found == 0) {
      intr_on();
      asm volatile("wfi");  // 没有可运行进程，等中断
    }
  }
}
```

scheduler 是每个 CPU 的无限循环，不是进程，没有 PID。遍历进程表找 RUNNABLE 的进程，通过 `swtch` 切换过去。

### scheduler 找到 init 进程

`userinit` 已经把 init 进程设为 RUNNABLE，scheduler 遍历时找到它，调用 `swtch(&c->context, &p->context)`。

## 六、第一个进程被调度的完整流程

### 第一步：swtch 切换到 init 进程

```asm
swtch:
        sd ra, 0(a0)     # 保存 scheduler 的寄存器到 c->context
        sd sp, 8(a0)
        ...
        ld ra, 0(a1)     # 从 p->context 恢复寄存器
        ld sp, 8(a1)     # sp = p->kstack + PGSIZE（内核栈）
        ...
        ret              # 跳到 ra = forkret
```

`allocproc` 设置了 `p->context.ra = forkret`，所以 `ret` 跳到 `forkret`。

### 第二步：forkret

```c
void forkret(void)
{
  static int first = 1;

  release(&myproc()->lock);

  if (first) {
    first = 0;
    fsinit(ROOTDEV);    // 第一个进程初始化文件系统
  }

  usertrapret();         // 准备返回用户态
}
```

新进程从未在用户态运行过，没有经历过"用户态 → trap → 内核态"。但回到用户态的唯一方式是通过 trampoline 的 `userret`。所以用 `forkret` → `usertrapret` → `userret` 这条路径，假装是从 trap 返回，实际上是第一次进入用户态。（参考 [新进程第一次被调度](020-xv6%20进程切换.md#六、新进程第一次被调度)）

### 第二步：initcode 执行

返回到用户态之后，initcode 执行 `exec("/init", argv)`，加载 `/init` 程序。`exec` 创建新页表，加载完整的 text、data、stack，丢弃旧页表。init 进程正式运行，它会创建控制台、启动 shell。

## 七、完整启动时间线

```
上电
  ↓
_entry（所有 CPU）
  ↓
start()（所有 CPU）
  ↓
main()
  ↓ CPU 0：kvminit → kvminithart → procinit → userinit → started=1
  ↓ 其他 CPU：等待 started==1 → kvminithart
  ↓
scheduler()（所有 CPU）
  ↓ CPU 0 的 scheduler 找到 init 进程
  ↓
swtch → forkret → usertrapret → userret → sret
  ↓
init 进程用户态：执行 initcode
  ↓
exec("/init") → 加载 /init 程序
  ↓
/init：创建控制台，fork + exec 启动 shell
  ↓
shell 等待用户输入
```

## 八、多核 CPU 的协调

- 只有 CPU 0 做全局初始化（`kvminit`、`userinit` 等）
- 其他 CPU 用 `while(started == 0)` 空转等待
- CPU 0 完成后设 `started = 1`，其他 CPU 各自安装内核页表后进入 scheduler
- 所有 CPU 独立运行 scheduler，从进程表中找可运行的进程
- `__sync_synchronize()` 是内存屏障，确保 `started = 1` 的写入对其他 CPU 可见

## 九、关键点总结

1. `main()` 不是被"调度"的，是上电后 CPU 直接执行到的
2. scheduler 不是进程，是每个 CPU 的"待机循环"
3. 第一个进程通过 `forkret → usertrapret → userret` 假装从 trap 返回，实际是第一次进入用户态
4. initcode 只做一件事：`exec("/init")`，真正的 init 程序从磁盘加载
5. `kvminithart` 开启分页是关键转折点——之前地址直接是物理地址，之后所有地址经过页表翻译
