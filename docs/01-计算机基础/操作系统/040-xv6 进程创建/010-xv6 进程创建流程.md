# xv6 进程创建流程

## 一、概述

xv6 中创建进程涉及多个函数，分工明确：

- `userinit`：创建第一个进程（`init`），内核启动时在 `main` 方法调用一次
- `allocproc`：分配进程基础设施（`PID`、`trapframe`、用户页表、内核上下文）
- `proc_pagetable`：创建用户页表，映射 `trampoline` 和 `trapframe`
- `fork`：复制当前进程创建子进程
- `exec`：用新程序替换当前进程的内存

## 二、allocproc — 进程的骨架

`allocproc` 在 `fork()` 和 `userinit()`（内核的`main()`运行到 `userinit()` 时已经开启MMU分页） 中被调用，负责初始化**一个**新进程的基础结构。

```c
static struct proc* allocproc(void){
  struct proc *p;

  // 1. 遍历进程表，找空闲 slot
  for(p = proc; p < &proc[NPROC]; p++) {
    acquire(&p->lock);
    if(p->state == UNUSED) {
      goto found;
    } else {
      release(&p->lock);
    }
  }
  return 0;

found:
  // 2. 分配 PID
  p->pid = allocpid();

  // 3. 分配 trapframe（一页物理内存，用于保存寄存器）
  if((p->trapframe = (struct trapframe *)kalloc()) == 0){
    release(&p->lock);
    return 0;
  }

  // 4. 创建用户页表（映射 trampoline 和 trapframe）  
  p->pagetable = proc_pagetable(p);
  if(p->pagetable == 0){
    freeproc(p);        // trapframe 已分配，需要回滚
    release(&p->lock);
    return 0;
  }

  // 5. 初始化内核上下文
  memset(&p->context, 0, sizeof(p->context));
  p->context.ra = (uint64)forkret;       // 第一次被调度时从 forkret 开始
  p->context.sp = p->kstack + PGSIZE;    // 内核栈底，内核栈的起始地址

  return p;
}
```

> [!TIP]
> 新进程从未被调度过，它的 `context` 是手动设置的，不是 `swtch` 保存的。
### 关键点

- `proc` 是全局数组 `struct proc proc[NPROC]`（64 个进程），`p = proc` 等价于 `p = &proc[0]`
- `kalloc()` 返回指针，因恒等映射可直接当物理地址使用（参考 [080-地址访问底层](../030-xv6%20页表/080-地址访问底层.md)）
- 错误处理遵循**逐步回滚**：第一个资源（trapframe）失败直接返回；第二个资源（页表）失败需调用 `freeproc` 释放已分配的 trapframe
- `allocproc` 只建骨架，不填充用户内存内容

## 四、三种进程创建方式

### 方式一：第一个进程（init）

```
main()
  → userinit()
    → allocproc()          // 空页表 + trampoline + trapframe
    → 手动映射一小段初始代码（initcode）
    → 设置 p->sz、trapframe 中的寄存器
    → p->state = RUNNABLE

init 进程被调度运行后：
  → initcode 执行 exec("init", argv)
    → 创建新页表 + trampoline + trapframe
    → 加载 /init 程序的 text、data、stack
    → 丢弃旧页表
```

参考[启动第一个进程](030-xv6%20启动第一个进程.md)
### 方式二：fork

```
fork()
  → allocproc()            // 空页表 + trampoline + trapframe
  → uvmcopy()              // 复制父进程的全部用户内存到子进程页表
  → 复制 trapframe（寄存器状态）
  → p->state = RUNNABLE
```

`uvmcopy` 逐页遍历父进程页表，对每个有效的叶子 PTE：`kalloc` 分配新物理页 → 复制内容 → `mappages` 映射到子进程页表的相同虚拟地址。子进程拥有父进程内存的完整副本。

### 方式三：fork + exec（最常见）

```c
if(fork() == 0) {
    exec("ls", argv);    // 子进程用新程序替换自己
}
```

fork 创建子进程（父进程的副本），exec 再用新程序替换：

```
exec("ls", argv)
  → 创建新页表（proc_pagetable）    // trampoline + trapframe
  → 读取 ELF 文件，逐段加载：
    → uvmalloc 分配物理页           // text 段
    → uvmalloc 分配物理页           // data 段
  → uvmalloc 分配两页              // guard page + stack
  → 在栈上设置 argc、argv
  → 丢弃旧页表（uvmfree）
  → 安装新页表到 p->pagetable
```

exec 创建新页表而不是修改旧页表，是**原子替换**策略：加载到一半失败时，旧页表不受影响，进程可继续运行。

## 五、进程销毁：资源释放顺序

进程退出时在 `freeproc` 中释放资源：

```
1. kfree(p->trapframe)              // 释放 trapframe 的物理页
2. uvmunmap(pt, TRAMPOLINE, 1, 0)   // 清 PTE，不释放物理页（共享的）
3. uvmunmap(pt, TRAPFRAME, 1, 0)    // 清 PTE，不释放物理页（已在步骤 1 释放）
4. uvmfree(pt, p->sz)               // 释放用户数据页 + 页表页
     → uvmunmap(pt, 0, sz/PGSIZE, 1)  // 清 PTE，释放用户数据的物理页
     → freewalk(pt)                    // 释放页表页本身
```

**为什么 trampoline 物理页不释放**

trampoline 代码在内核代码段里，所有进程共享同一个物理页。释放了别的进程就崩了。

**为什么 trapframe 需要单独释放**

`uvmfree` 只释放 0 到 `p->sz` 范围的页，TRAPFRAME 不在这个范围内。`uvmunmap` 清 PTE 时最后一个参数为 0（不释放物理页），所以必须在之前单独 `kfree`。

**为什么必须先清叶子 PTE 再 freewalk**

`freewalk` 遍历页表时，如果遇到有效的叶子 PTE（有 R/W/X 权限位），会 `panic("freewalk: leaf")`，因为这意味着物理页泄漏。所以必须先把所有叶子映射清干净。


## 六、内核上下文初始化

```c
memset(&p->context, 0, sizeof(p->context));
p->context.ra = (uint64)forkret;
p->context.sp = p->kstack + PGSIZE;
```

- `context` 存放 callee-saved 寄存器（ra、sp、s0-s11）
- `ra` 设为 `forkret`：新进程第一次被 scheduler 通过 `swtch` 调度时，`ret` 跳到 `forkret`
- `sp` 设为内核栈顶（栈从高地址向低地址增长）
- `forkret` 会调用 `usertrapret()`，最终通过 trampoline 返回用户态

新进程第一次被调度的完整流程，详见 [xv6 进程切换 - 第八节](020-xv6%20进程切换.md#八、新进程第一次被调度)。
