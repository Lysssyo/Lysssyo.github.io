# mihomo 链式代理笔记

## 1. 链式代理是什么

链式代理就是让流量经过多个代理节点后再访问目标网站。

普通代理：

本机程序 → mihomo → 代理节点 → 目标网站

链式代理：

本机程序 → mihomo → 节点A → 节点B → 目标网站

如果节点 B 是静态 IP 出口，那么最终目标网站看到的 IP 通常是 **节点 B 的出口 IP**。

---

## 2. mihomo 中用 `relay` 实现链式代理

你当前配置里有：

proxy-groups:  
- name: Relay-StaticIP  
  type: relay  
  proxies:  
  - 🇦🇱Albania  
  - 我的静态IP出口

含义是：

本机 → mihomo → 🇦🇱Albania → 我的静态IP出口 → 目标网站

其中：

🇦🇱Albania       = 第一跳节点  
我的静态IP出口   = 第二跳节点 / 最终出口

顺序很重要。

---

## 3. `relay` 的顺序含义

配置：

proxies:  
  - 🇦🇱Albania  
  - 我的静态IP出口

表示：

先连 🇦🇱Albania  
再通过 🇦🇱Albania 去连 我的静态IP出口  
最后由 我的静态IP出口 访问目标网站

所以最终网站看到的出口 IP，一般应该是：

我的静态IP出口 的 IP

不是 `🇦🇱Albania` 的 IP。

---

## 4. `Relay-StaticIP` 和 `Proxy` 的关系

你还有一个选择组：

- name: Proxy  
  type: select  
  proxies:  
  - Relay-StaticIP  
  - ♻️ 自动选择  
  - 🎯 全球直连  
  - 🇭🇰丛雨云  
  - 🇯🇵全部超时请检查系统时间  
  - 🇸🇬请开启自动更新订阅  
  - 🇹🇼congyu.org 🐱  
  - 🇭🇰Hong Kong ムラサメ  
  - 🇭🇰x0.7 倍率 测试节点  
  - 🇯🇵Japan 春日野 悠  
  - 🇯🇵Japan 春日野 穹

含义是：

Proxy 是一个可选择策略组  
Relay-StaticIP 是 Proxy 里面的一个选项

也就是说，规则如果写的是：

- MATCH,Proxy

那么流量会先进入 `Proxy` 组。

如果 `Proxy` 当前选择的是：

Relay-StaticIP

那么最终流量才会走链式代理。

---

## 5. 判断是否真的用到了链式代理

需要同时满足两个条件：

1. rules 规则把流量指向 Proxy 或 Relay-StaticIP  
2. 如果指向 Proxy，则 Proxy 当前选择的是 Relay-StaticIP

例如：

rules:  
  - MATCH,Proxy

并且 `Proxy` 当前选中：

Relay-StaticIP

那么实际链路是：

本机 → mihomo → 🇦🇱Albania → 我的静态IP出口 → 目标网站

如果 `Proxy` 当前选中的是：

♻️ 自动选择

那就不会走 `Relay-StaticIP`。

---

## 6. 查看两个节点具体是什么

查看 `🇦🇱Albania` 节点：

awk '/name:.*🇦🇱Albania/{flag=1;count=0} flag{print;count++} count==25{flag=0}' /root/mihomo/config.yaml

查看 `我的静态IP出口` 节点：

awk '/name:.*我的静态IP出口/{flag=1;count=0} flag{print;count++} count==25{flag=0}' /root/mihomo/config.yaml

输出里可能会有敏感字段，例如：

server:  
port:  
uuid:  
password:  
cipher:

不要随便公开贴出完整内容。

---

## 7. 查看 `Relay-StaticIP` 策略组

awk '/name: Relay-StaticIP/{flag=1;count=0} flag{print;count++} count==20{flag=0}' /root/mihomo/config.yaml

正常应该看到类似：

- name: Relay-StaticIP  
  type: relay  
  proxies:  
  - 🇦🇱Albania  
  - 我的静态IP出口

---

## 8. 查看规则有没有直接用到 `Relay-StaticIP`

grep -n "Relay-StaticIP" /root/mihomo/config.yaml

如果它只出现在：

proxy-groups:

里，而没有出现在：

rules:

里，说明规则不是直接指向 `Relay-StaticIP`。

这种情况下要看规则是否指向了 `Proxy`。

---

## 9. 查看规则有没有用到 `Proxy`

grep -n ",Proxy" /root/mihomo/config.yaml

重点看最后有没有：

- MATCH,Proxy

也可以直接看规则末尾：

tail -50 /root/mihomo/config.yaml

最后的 `MATCH` 很关键，例如：

- MATCH,Proxy

表示前面规则都没匹配时，默认走 `Proxy`。

如果是：

- MATCH,🎯 全球直连

表示默认直连，不走代理。

---

## 10. 查询当前运行时 `Proxy` 选中了什么

无 secret 时：

curl http://127.0.0.1:9090/proxies/Proxy

如果配置了 secret：

curl -H "Authorization: Bearer 你的secret" http://127.0.0.1:9090/proxies/Proxy

看返回里的：

"now":"Relay-StaticIP"

如果 `now` 是 `Relay-StaticIP`，说明 `Proxy` 当前选中了链式代理。

如果 `now` 是其他节点，例如：

"now":"♻️ 自动选择"

那流量不会通过 `Relay-StaticIP`。

---

## 11. 手动把 `Proxy` 切换到 `Relay-StaticIP`

无 secret 时：

curl -X PUT http://127.0.0.1:9090/proxies/Proxy \  
  -H "Content-Type: application/json" \  
  -d '{"name":"Relay-StaticIP"}'

有 secret 时：

curl -X PUT http://127.0.0.1:9090/proxies/Proxy \  
  -H "Authorization: Bearer 你的secret" \  
  -H "Content-Type: application/json" \  
  -d '{"name":"Relay-StaticIP"}'

然后再检查：

curl http://127.0.0.1:9090/proxies/Proxy

---

## 12. 测试最终出口 IP

使用 mihomo 的 HTTP 代理端口测试：

curl -x http://127.0.0.1:7890 https://ipinfo.io/ip

不走代理测试：

curl https://ipinfo.io/ip

如果走代理后的 IP 是你的静态 IP 出口，说明链式代理成功。

也可以测试：

curl -x http://127.0.0.1:7890 https://ifconfig.me

---

## 13. 修改配置后检查和重启

先检查配置语法：

/root/mihomo/mihomo -t -d /root/mihomo

如果没问题，再重启服务：

systemctl restart mihomo

查看状态：

systemctl status mihomo

查看日志：

journalctl -u mihomo -n 50 --no-pager

## 14. 常见问题

### 问题 1：配置了 `Relay-StaticIP`，但实际没走链式代理

可能原因：

rules 没有指向 Proxy 或 Relay-StaticIP  
Proxy 当前没有选中 Relay-StaticIP  
前面的规则提前匹配到了 DIRECT 或其他策略组  
节点本身不可用

---

### 问题 2：最终 IP 不是静态 IP

可能原因：

Relay 顺序写反了  
Proxy 没有选中 Relay-StaticIP  
rules 没有走到 Proxy  
我的静态IP出口节点不可用  
请求没有走 127.0.0.1:7890 代理

---

### 问题 3：链式代理很慢

正常。因为链路变成：

本机 → 第一跳 → 第二跳 → 目标网站

中间多了一跳，延迟和丢包都会更明显。

---

### 问题 4：UDP 不稳定

`relay` 通常主要适合 TCP。  
对于游戏、QUIC、部分 UDP 应用，链式代理可能效果不好。
