# “翻墙”基本原理

当我们在 Chrome 输入网址 `www.google.com`，按下回车，这个请求到底是怎么翻越 GFW 到达 Google 的服务器的呢？



分步讲解

我们将剥离掉所有复杂的上层概念，只看底层 Socket 和协议的交互过程：

第一步：本地握手与劫持 (Local Handshake)

当你在 Chrome 输入网址按下回车时，浏览器发现系统设置了代理。它不会自己去广阔的互联网上找 Google，而是直接向你本机的 Clash（例如 127.0.0.1:7890）发起第一次 TCP 三次握手。

连上后，Chrome 发送一条纯文本指令：`CONNECT www.google.com:443。`意思是：“Clash，请帮我把后续的流量接通到 Google 的 443 端口”。

> [!important] connect 方法
> `CONNECT` 是 HTTP 协议中的一个特殊方法（Method），专门用于代理服务器环境。它的核心作用是要求代理服务器**建立一条端到端的透明盲传隧道（Transparent Blind Relay Tunnel）**。
> 当浏览器需要访问 HTTPS（端口 443）等加密服务时，代理服务器无法解密中间的流量。因此，浏览器必须先用明文的 `CONNECT` 指令告诉代理目标地址，代理接通后，双方再开始传输加密数据。

在这个 `CONNECT www.google.com:443` 阶段，具体解决了以下几个关键问题：

- **解决“代理无法解密 HTTPS”的困境：** 如果是传统的明文 HTTP 时代，浏览器直接把 `GET http://www.google.com/` 扔给代理就行，代理能看懂并代发。但现代网络都是 HTTPS 加密的，浏览器生成的数据包已经是基于 TLS 加密的密文。Clash 没有 Google 的私钥，根本无法从这堆密文里找出你要访问的域名。所以，浏览器必须在发送密文之前，先用明文的 `CONNECT` 方法向代理“报备”目的地。
- **角色转换（从 HTTP 代理到 TCP 盲转发）：** `CONNECT` 指令本质上是告诉 Clash：“不要尝试解析接下来的 HTTP 报文了”。当 Clash 把远程连接打通，并向浏览器回复 `HTTP/1.1 200 Connection Established` 之后，它在这个连接上的角色就彻底变了。它不再是一个 HTTP 代理，而变成了一个“透明的 TCP 管道”。浏览器发来的任何 TCP 包，Clash 都会闭着眼睛原封不动地塞进隧道发往远端；远端传回来的数据，也会原封不动地交回给浏览器。
- **承载 TLS 握手：** 只有在 `CONNECT` 指令成功执行，这条透明管道搭建完毕后，你的 Chrome 才会真正开始向里面发送用于建立 HTTPS 安全连接的 `TLS Client Hello` 数据包
    

第二步：建立跨境代理隧道 (Proxy Tunnel Handshake)

Clash 收到指令，查阅规则后决定把任务交给你的代理节点，例如 🇸🇬 新加坡-中转 04 节点。

Clash 此时作为客户端，向这台新加坡服务器的公网 IP 发起第二次 TCP 三次握手。TCP 连通后，立刻进行一层 TLS 握手（如果是 Trojan 或 Vmess over TLS 协议）。

> [!TIP]
> 这里的 TLS 握手是为了建立一层类似 HTTPS 的加密外壳，但其内部包裹的不是普通的网页数据，而是代理协议（如 Trojan 或 Vmess）。GFW 之所以不拦截这次 TCP 和 TLS 握手，是因为它在行为特征上**完全伪装成了普通网民对海外正常网站的一次合法 HTTPS 访问**。

> [!important] 代理协议
> [代理协议](代理协议.md)

**表面上（给 GFW 看的）：** 现代的代理协议（如 Trojan 或 Vmess over TLS）的设计哲学就是“打不过就加入”。机场会在新加坡服务器上搭建一个真实的、可以访问的伪装网站（比如一个英文博客），并申请真实的域名和 TLS 证书。

**本质上（对 Clash 而言）：不是普通的 HTTPS。** 它是将代理协议指令（例如之前提到的 `CONNECT www.google.com`）作为数据，塞进了 TLS 这条安全管道里。这叫做“TLS 隧道承载代理流量”。

**问题二：为什么这个 TCP 握手和 TLS 握手不会被墙？** GFW 是一个极其庞大且复杂的流量过滤系统，它的封锁是基于**成本和规则**的：

1. **TCP 握手不被墙，是因为 IP 是干净的：** 互联网每天有海量的跨国 TCP 连接。只要你连接的这个新加坡公网 IP **目前不在 GFW 的黑名单里**，GFW 就会默认允许建立底层的 TCP 连接。墙不会无缘无故阻断所有出国的 TCP 包，否则整个跨国互联网就瘫痪了。
2. **TLS 握手不被墙，是因为它看起来太“正常”了：** 全球互联网的金融、购物、社交几乎 100% 都在使用 TLS（HTTPS 的底层加密）。GFW **不可能也绝不敢**屏蔽所有跨境的 TLS 握手。
    - 在图中的 **Step B**，Clash 发出的 `TLS Client Hello` 中包含一个明文的 SNI（服务器名称指示），写着你要访问的那个伪装域名。GFW 看了看：域名没在黑名单，格式是标准的 TLS 1.3，服务器回传的证书也是国际知名机构颁发的真证书。
    - 于是 GFW 的逻辑判定：**“这是一个中国网民正在访问一个合法的新加坡英文博客，放行。”**
3. **后续加密流量无法被识别：** 一旦 TLS 握手完成（**Step C**），后续所有的代理指令和翻墙数据全变成了高强度的随机乱码（密文）。GFW 没有私钥，解不开这层外衣，它只能看到两端在进行正常的“网页数据传输”，从而实现安全翻墙。

第三步：远端目标握手 (Remote Target Handshake)

在建立好的加密隧道内，Clash 悄悄把 Chrome 的诉求发了过去。

新加坡服务器解开外层加密后明白：“客户要连 Google”。于是，它代替你的电脑，在新加坡当地的机房向 Google 的边缘节点发起第三次 TCP 三次握手。

连通后，新加坡服务器顺着隧道告诉 Clash：“接通了！”；Clash 再转告 Chrome：“管道搭好了，你发数据吧”。

第四步：端到端真实握手 (End-to-End Handshake)

Chrome 认为自己已经直连了 Google，于是开始发送自己的 TLS Client Hello 去和 Google 协商网页的加密密钥。

这个包极其安全，因为它被加密了两次：内层是 Chrome 与 Google 之间的 HTTPS 加密，外层是 Clash 与机场之间的隧道加密。新加坡节点只负责将这个加密包原封不动地转发给 Google。

TLS 握手完成后，Chrome 正式发出 GET / 请求，Google 的搜索首页代码顺着这条双重加密的管道，跨越物理大洋，最终呈现在你的屏幕上。

在这套极其严密的流程中，你可能会好奇：Chrome 在第一步连 Clash 之前，连 IP 都不知道，是怎么把` www.google.com `变成网络包发出来的？需要我为你揭秘 Clash 网络堆栈里最核心、也是最能提升访问速度的黑科技——Fake-IP（伪装 IP）机制吗？
