# 代理协议

在标准的 HTTPS 访问中，TLS 的上层确实是 **HTTP 协议**。

![image.png](https://keith-knowledge-base.oss-accelerate.aliyuncs.com/20260310144845836.png)


但在使用翻墙代理（如 Trojan、Vmess over TLS、VLESS）时，这层外层 TLS 握手完毕后，它的上层**真实协议是“代理协议本身”**（Proxy Protocol），而**伪装协议是“HTTP/2 或 HTTP/1.1”**。

- **Trojan（代理协议）：** 属于**一次性路由寻址协议**。它的头部（包含密码哈希和目标地址）只在每个 TCP Socket 连接建立的**最开始**发送一次。寻址完成后，协议本身“隐身”，剩下的生命周期完全变成纯粹的 TCP 字节流盲传（包裹在外层 TLS 中）。
- **HTTPS（HTTP over TLS）：** 属于**事务型传输协议**。即使底层的 TCP/TLS 连接保持不断开（Keep-Alive），你发送的**每一次**请求（GET、POST 等），都必须带有完整的 HTTP 头部。HTTP 协议贯穿于整个连接的生命周期。

当 Clash 和海外节点 TLS 握手完成后，隧道建立。此时在这个隧道里流淌的数据，面临着“表里不一”的情况：

**第一层：给 GFW 看的“伪装上层”（ALPN 协商）** 在 TLS 握手阶段，有一个专门的扩展字段叫 **ALPN**（应用层协议协商）。 Clash 在发出 `TLS Client Hello` 时，会在 ALPN 里明文写上一句：“服务器你好，握手成功后，我们上层用 **h2 (HTTP/2)** 或者 **http/1.1** 来通讯吧。” GFW 抓到这个包，一看 ALPN 是 `h2`，就会在它的记录本上写下：**这个 TLS 的上层是 HTTP/2 协议**。

**第二层：真实的“物理上层”（代理协议）** TLS 握手一旦完成，双方进入加密通讯阶段，GFW 就瞎了。此时，Clash 和新加坡节点会立刻“翻脸不认人”，它们根本不会发送任何符合 HTTP 规范的 `GET` 或 `POST` 请求。 它们在 TLS 之上运行的真实协议，是 **Trojan** 或 **Vmess** 等代理协议。

以 Trojan 为例，TLS 握手完成后的**第一个加密数据包**，根本不是 HTTP 头部，而是极其精简的 Trojan 协议头：

1. **56 字节的密码哈希**。
2. **回车换行符**（CRLF）。
3. **目标地址指令**（比如包含 `www.google.com:443`，告诉节点要把剩下的数据转发给谁）。
4. **回车换行符**（CRLF）。
5. **真实的用户数据**（Chrome 发给 Google 的内层 TLS Client Hello）。
    

**总结：** 对于 GFW 而言，这层 TLS 的上层是它猜测的 **HTTP 网页协议**； 对于 Clash 和机场节点而言，这层 TLS 的上层是 **Trojan / Vmess 代理协议**； 而代理协议内部，又包裹着你真实访问 Google 的流量。这就构成了一套完美的“俄罗斯套娃”网络模型。

## Trojan 包结构

### 1. 核心概念定义

Trojan 协议的设计哲学是 **“极简与克制”** 。它没有自己复杂的加密算法，而是完全将自己寄生在标准的 TLS 加密隧道内。

当外层的Clash客户端 和 海外节点 TLS 握手完成后，你的 Clash 发出的**第一个加密数据包**，就是 Trojan 的请求头。这个请求头本质上是一段极其紧凑的**内存字节块（Raw Bytes）**，它直接拼接在你要发送的真实业务数据（比如 Chrome 发给 Google 的 TLS Client Hello）之前。

### 2. 可视化表示（字节结构布局）

针对这种底层协议的内存和字节布局，我们使用 ASCII 结构图来精确剖析：

```
+---------------------------------------------------------------+
|                  Password Hash (56 Bytes)                     |
|          (SHA224 of your proxy password, in hex)              |
+---------------------------------------------------------------+
|                        CRLF (2 Bytes)                         |
|                            "\r\n"                             |
+---------+----------+----------------------+-------------------+
| CMD (1) | ATYP (1) | DST.ADDR (Variable)  | DST.PORT (2 Bytes)|
| 0x01/02 | 0x01/3/4 |   e.g., google.com   |    e.g., 443      |
+---------+----------+----------------------+-------------------+
|                        CRLF (2 Bytes)                         |
|                            "\r\n"                             |
+---------------------------------------------------------------+
|                       Payload (Variable)                      |
|       (e.g., Chrome's raw TLS Client Hello to Google)         |
+---------------------------------------------------------------+
```

### 3. 分步讲解

我们可以把这个包拆解为三个核心部分来理解它的精妙之处：

- **身份验证层 (Password Hash + CRLF)：**
    
    前 56 个字节是你机场节点密码的 SHA224 哈希值（以十六进制字符串表示），紧接着是一个回车换行符 `\r\n`。
    
    **防御 GFW 的关键就在这里：** 当 GFW 的主动探测器发来随机乱码时，Trojan 服务端会读取前 56 个字节。如果哈希对不上，Trojan 会立刻判定“这不是我的客户端”。此时，它**不会**断开连接，而是将这 56 个字节以及后面的所有探测乱码，原封不动地转发给背后的 Nginx。
    
- **寻址指令层 (CMD + ATYP + DST.ADDR + DST.PORT)：**
    - **CMD (1 Byte):** 指令类型。`0x01` 代表 TCP 连接，`0x02` 代表 UDP 转发。
    - **ATYP (1 Byte):** 目标地址类型。`0x01` 是 IPv4，`0x03` 是域名，`0x04` 是 IPv6。
    - **DST.ADDR (Variable):** 目标地址。如果是访问 Google，这里就是 `www.google.com` 的字节码。
    - **DST.PORT (2 Bytes):** 目标端口。比如 443 端口。
- **数据隔离与载荷 (CRLF + Payload)：**
    
    第二个 `\r\n` 是一个极其重要的**分隔符**。它告诉 Trojan 服务端：“代理的指令到此结束，接下来的所有字节，全是用户要发送给目标网站的真实数据（Payload）”。
    
    一旦 Trojan 成功解析了这个头部，并在新加坡当地为你和 Google 建立了 TCP 连接，这个 Trojan 头部就会被剥离丢弃。此后，这条隧道就变成了一条纯粹的透明水管，只负责**盲传**你的 Payload。（在海外节点成功用 TCP 连接上 Google 的那一毫秒，Trojan 协议的使命就结束了。那个包含密码哈希和目标域名的“头部（Header）”从海外节点的内存中被永久删除（剥离丢弃）**剥离丢弃后，具体发生了什么？（代码层面的盲传）** 一旦头部被丢弃，海外节点的内存里就只剩下两个极其重要的东西：

1. **连接 A（Socket A）：** 和你本地 Clash 保持连通的 TLS 加密隧道。
2. **连接 B（Socket B）：** 刚刚和 Google 建立好的 TCP 连接。
    

此时，海外节点上的代理程序会启动两个非常简单的“死循环（Infinite Loop）线程”：

- **上行线程（Upload）：** 只要从 **连接 A**（你的电脑）读到任何一个字节，立刻原封不动地塞进 **连接 B**（发给 Google）。它绝对不看这个字节是 HTTP 代码还是图片数据。
- **下行线程（Download）：** 只要从 **连接 B**（Google）读到任何一个字节，立刻用外层 TLS 加密，然后原封不动地塞进 **连接 A**（发回给你本地的 Clash）。）

所以说，Trojan协议只在建立连接

通过这种结构，你在本地发出的 `CONNECT www.google.com:443` 指令，就被完美地序列化成了这几十个字节的二进制串，塞进了 TLS 隧道里。


这是一个**“全链路端到端跨国代理生命周期图”**。它将**网络协议栈的嵌套**（OSI 模型的三跳剥离）、**GFW 的审查节点**，以及代理服务器核心的**状态机转换**（从协议解析器到透明水管）完美地融合在了一条时间线上：

![未命名绘图.drawio.png](https://keith-knowledge-base.oss-accelerate.aliyuncs.com/%E6%9C%AA%E5%91%BD%E5%90%8D%E7%BB%98%E5%9B%BE.drawio.png)

这张合并后的图清晰地揭示了这套“魔术”的完整运作机制：

1. **Hop A（本地启航）：** Chrome 通过本地网络栈向 Clash 发出明文的 `CONNECT` 指令，申请一条前往 Google 的通道。
2. **Hop B（穿越 GFW 的伪装与运送）：** Clash 将请求包装在 Trojan 协议中，并在最外层套上 TLS 1.3 的装甲。GFW 在中间进行深层包检测（DPI），但只看到了合法的 TCP 和 TLS 握手，于是将这股包含了“Trojan 头部 + 真实 Chrome 数据”的加密洪流放行到了海外节点。
3. **生死交界（状态切换）：** 海外节点收到数据包，剥开外层 TLS，读取 Trojan 头部并验证密码。确认你要去 Google 后，它立刻代替你向 Google 发起第三次 TCP 握手（Hop C）。**就在握手成功的那一瞬间，节点将 Trojan 头部彻底销毁，自身化作一根透明的“盲传水管”。**
4. **无缝对接（内层 TLS 握手）：** 管道打通后，原本被 Trojan 头部堵在后面的 `Chrome Payload`（也就是 Chrome 发出的真实 `Inner TLS Client Hello`）终于被冲刷进了 Google 的服务器。
5. **端到端循环（The Blind Relay Loop）：** 从此刻起，Chrome 和 Google 开始了真正的“双向奔赴”。对于 Google 而言，它以为自己正和新加坡的一台普通电脑进行 HTTPS 加密通讯；对于 Chrome 而言，它以为自己直接连上了 Google。在这条坚不可摧的隧道里，数据开始了高速的、纯粹的 I/O 拷贝。

到目前为止，我们探讨的都是基于 TCP 的网页浏览。但如果你通过代理打游戏，或者进行 DNS 解析，就必须用到 UDP 协议。由于原始的 TLS 只能承载 TCP 流，你知道现代代理协议（比如这里的 `CMD: 0x02`）是如何实现**“在 TCP/TLS 隧道里强行塞入 UDP 数据包”**（UDP over TCP）的吗？
