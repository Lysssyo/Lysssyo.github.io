# 图书馆预约控制台设计

## 背景

`Lysssyo/library-reservation` 是一个广州大学图书馆自动预约系统，通过阿里云函数计算（FC）定时触发，调用 GitHub Actions 执行 Playwright 自动化脚本完成预约。

日常使用中需要：
1. **临时关闭/开启定时触发器**（如不需要预约时）
2. **更换预约的座位号**（通过修改 GitHub Actions Variable `LIB_SEAT_ID`）

原始方式是手动进阿里云控制台和 GitHub 设置页操作，不够方便。因此在 VitePress 知识库里新增一个控制台页面，实现一键操作。

---

## 整体架构

```
VitePress 页面（LibraryControl.vue）
    ↓ HTTP POST（JSON）
阿里云 FC HTTP 函数（library-reservation-ctrl）
    ├─ action: status  → 查询 timer/timer2 触发器状态 + GitHub Variable LIB_SEAT_ID
    ├─ action: toggle  → 同时开/关 timer 和 timer2 两个触发器
    └─ action: set-seat → 更新 GitHub Actions Variable LIB_SEAT_ID
```

---

## 核心设计决策：为什么用 FC 转发，而不是前端直接调用

这是整个方案里最关键的架构决策。

### 方案一：前端直接调用阿里云 FC 管理 API（❌ 不可行）

阿里云 FC 管理 API（如 `UpdateTrigger`、`GetTrigger`）是**服务端 API**，设计上不支持浏览器调用，原因有两点：

**1. CORS 限制（技术硬伤）**

阿里云 FC 管理 API 的域名（`*.fc.aliyuncs.com`）不返回 `Access-Control-Allow-Origin` 响应头，浏览器的同源策略会直接拦截请求，无论如何配置前端代码都无法绕过。

**2. 签名算法（安全硬伤）**

阿里云 API 每次请求都需要用 `AccessKey Secret` 做 HMAC-SHA256 签名。如果在前端执行签名，就必须把 AK/SK 写入 JavaScript bundle，任何打开 DevTools 的人都能读取，进而操控整个阿里云账号。

### 方案二：前端直接调用 GitHub API（⚠️ 部分可行）

GitHub REST API 支持 CORS，前端可以直接调用（如更新 Variable）。但同样面临凭据暴露问题——GitHub PAT 会出现在 JS bundle 里。

对于个人使用、不对外公开的页面，风险可接受，但需要在前端维护两套凭据（阿里云 AK/SK + GitHub PAT），且阿里云部分因 CORS 问题仍然行不通。

### 方案三：FC HTTP 函数作为代理（✅ 最终方案）

参考知识库中「私有保险箱」的设计模式——`PrivateVault.vue` 调用一个 FC HTTP 函数作为代理，敏感凭据存在 FC 函数的环境变量里，不暴露给浏览器。

本方案完全复用这一模式：
- **前端**只持有 FC HTTP 触发器的公开 URL，无任何密钥
- **FC 函数环境变量**存储 `ALIBABA_CLOUD_ACCESS_KEY_ID`、`ALIBABA_CLOUD_ACCESS_KEY_SECRET`、`GITHUB_PAT`
- FC 函数在服务端完成签名和 API 调用，返回结果给前端

| | 方案一（直调阿里云） | 方案二（直调 GitHub） | 方案三（FC 转发） |
|---|---|---|---|
| CORS | ❌ 被拦截 | ✅ 支持 | ✅ 服务端无限制 |
| 凭据安全 | ❌ AK/SK 暴露 | ⚠️ PAT 暴露 | ✅ 凭据在服务端 |
| 阿里云操作 | ❌ | ❌ | ✅ |
| GitHub 操作 | ❌ | ✅ | ✅ |

## FC 函数实现

**函数名**：`library-reservation-ctrl`
**运行时**：Python 3.x
**触发器**：HTTP 触发器
**入口**：`index.handler`

### Handler 签名

调试过程中发现阿里云 FC 的实际 handler 签名为 `def handler(event, context)`，`event` 是 `bytes` 类型。HTTP 请求的完整信息（method、headers、body）被封装为 JSON 后作为 bytes 传入：

```json
{
  "version": "v1",
  "rawPath": "/",
  "headers": { ... },
  "queryParameters": {},
  "body": "{\"action\": \"status\"}",
  "isBase64Encoded": false,
  "requestContext": {
    "http": {
      "method": "POST",
      ...
    }
  }
}
```

因此解析方式为：
```python
event_data = json.loads(event.decode('utf-8'))
method = event_data['requestContext']['http']['method']
body = json.loads(event_data['body'])
```

### 响应格式

```python
return {
    'statusCode': 200,
    'headers': { 'Access-Control-Allow-Origin': '*', ... },
    'body': json.dumps(result),
}
```

### 环境变量

| 变量名 | 用途 |
|--------|------|
| `ALIBABA_CLOUD_ACCESS_KEY_ID` | 阿里云 AK，用于调用 FC 管理 API |
| `ALIBABA_CLOUD_ACCESS_KEY_SECRET` | 阿里云 SK |
| `GITHUB_PAT` | GitHub Personal Access Token，需要目标仓库的 `repo` 权限（用于读写 Actions Variables） |

### 三个接口

**status**：查询当前状态
- 调用 `GetTrigger` 读取 `timer` 触发器的 `trigger_config`，解析 `enable` 字段
- 调用 GitHub API `GET /repos/{owner}/{repo}/actions/variables/LIB_SEAT_ID` 读取座位号
- 两个触发器（`timer`、`timer2`）状态一致，读一个即可

**toggle**：开/关触发器
- 循环对 `timer` 和 `timer2` 调用 `UpdateTrigger`，写入 `{"enable": true/false}`
- 两个触发器必须同时操作，保证状态一致

**set-seat**：更新座位号
- 调用 GitHub API `PATCH /repos/{owner}/{repo}/actions/variables/LIB_SEAT_ID`
- 成功返回 HTTP 204

## 前端实现

### 座位号映射

图书馆 API 使用的是内部座位 ID（如 `101267824`），对用户不友好。实际座位编号是一个较小的数字，与内部 ID 的差值固定为 `101267703`。

前端做双向映射：
- **展示**：`display = storedId - 101267703`
- **提交**：`storedId = input + 101267703`

GitHub Variable 和 FC 函数始终存储完整 ID，映射只在前端 UI 层处理。

### 页面结构

`LibraryControl.vue` 在 `onMounted` 时调用 `status` 接口初始化状态，包含：
- 触发器开关按钮（显示当前状态，点击取反并调用 `toggle`）
- 座位号输入框（显示映射后的简短编号，提交时换算回完整 ID）
- 操作结果提示（3 秒后自动消失）

### 路由配置

`docs/library.md` 作为页面入口，加入 `excludeByGlobPattern` 避免出现在知识库侧边栏，同时在顶部导航栏新增「图书馆预约」入口。

## 踩坑记录

### 1. 全角空格污染导致文件编辑失败

从阿里云控制台复制的示例代码中混入了全角空格（`\u3000`）而非普通空格，Edit 工具按字符串精确匹配，导致始终找不到目标内容报错。排查方式是用 `cat -A` 查看原始字节，发现 `M-BM-` 前缀（UTF-8 编码的全角空格）。最终用 Write 工具整体重写文件解决。

### 2. FC 依赖问题：`pip install` 不生效

在 FC 控制台终端执行 `pip install alibabacloud_fc20230330` 后，函数仍报 `ModuleNotFoundError`。原因是 FC 运行时环境与终端的 Python 环境是隔离的，`pip install` 安装到了终端的 site-packages，函数执行时找不到。

尝试 `pip install -t . alibabacloud_fc20230330` 把包安装到当前目录（与 `index.py` 同级），结果把大量包文件散落在函数根目录，难以清理。最终通过 FC 控制台的「层（Layer）」功能或内置依赖方式解决，并清空根目录重新部署。

### 3. Handler 签名错误：WSGI vs event/context

最初参考通用 Python HTTP 服务的写法，用了 WSGI 风格：

```python
def handler(environ, start_response):
    if environ.get('REQUEST_METHOD') == 'OPTIONS':
```

点击「测试函数」后报错：

```
'bytes' object has no attribute 'get'
```

原因是阿里云 FC 的入口**只支持** `def handler(event, context)`，`event` 是 `bytes`。点「测试函数」走的是事件触发接口，直接把 payload 作为 bytes 传入，`environ` 实际上是 bytes 对象，调 `.get()` 自然报错。

通过写一个调试版 handler 打印 event 的类型和内容，发现 HTTP 请求被封装成了如下 JSON bytes：

```json
{
  "version": "v1",
  "rawPath": "/",
  "headers": { ... },
  "body": "{\"action\": \"status\"}",
  "requestContext": {
    "http": { "method": "POST", ... }
  }
}
```

据此修正解析方式，正式代码改为：

```python
def handler(event, context):
    event_data = json.loads(event.decode('utf-8'))
    method = event_data['requestContext']['http']['method']
    body = json.loads(event_data['body'])
```

> 注意：「测试函数」按钮不适合测试 HTTP 触发函数，应通过 HTTP 触发器 URL 用 curl 或 Apifox 测试。

### 4. GITHUB_PAT 环境变量未读到

`status` 接口返回 `seatId: ""`，加日志后发现：

```
GitHub API status: 401, body: {"message": "Bad credentials"}
```

进一步打印 token 长度：

```python
token = os.environ.get('GITHUB_PAT', '')
logger.info(f"GITHUB_PAT length: {len(token)}, prefix: {token[:6]}")
```

输出 `length: 0`，说明环境变量根本没读到。检查 FC 控制台发现环境变量一栏为空——部署时漏填了。在 FC 控制台 → 函数配置 → 环境变量 中补填 `GITHUB_PAT` 后恢复正常。
